# Plano: Tornar a Extração de Vector Memory Inteligente (só quando vale a pena)

## Contexto

O sistema de vector memory (`vector_memory_enabled=True`) dispara extração de memórias em **TODA interação**, fazendo 4 chamadas LLM por turno (3 para correções/preferências/fatos + 1 para self-correction). Conversas triviais como "oi", "obrigado", "qual o horário?" gastam tokens desnecessariamente.

**Trigger points atuais (todos disparam incondicionalmente):**
- `backend/app/worker/tasks.py:392` — `extract_and_save_memories()`
- `backend/app/worker/tasks.py:1626` — `extract_agent_self_corrections()`
- `backend/app/orchestrator/supervisor.py:454` — `extract_and_save_memories()`
- `backend/app/orchestrator/gate.py:338` — `extract_and_save_memories()`

## Abordagem: Pré-filtro Heurístico Barato

Adicionar uma função de gating rápida que decide se vale a pena chamar o LLM de extração. O pré-filtro usa heurísticas baratas (sem LLM) para descartar interações triviais.

### Mudança 1: Novo módulo `backend/app/services/memory_gate.py`

Criar função `should_extract_memories(user_message, agent_response, history) -> bool` que retorna `False` para interações triviais.

**Heurísticas (baratas, sem LLM):**

1. **Tamanho mínimo** — Se `user_message` tem menos de ~15 caracteres e não contém palavras-chave de correção/preferência, pular. Ex: "ok", "oi", "obrigado", "tchau", "sim", "não".

2. **Lista de mensagens triviais** — Padrões comuns que nunca geram memória útil:
   - Saudações isoladas: "oi", "olá", "bom dia", "boa tarde", "boa noite", "e aí", "opa"
   - Agradecimentos: "obrigado", "valeu", "brigado", "thanks"
   - Confirmações curtas: "ok", "tá", "certo", "beleza", "entendi", "sim", "não", "pode ser"
   - Despedidas: "tchau", "até mais", "até logo", "flw"
   - Menos de 3 palavras no geral

3. **Palavras-chave de alta prioridade (sempre extrair)** — Se a mensagem contém indicadores de correção ou preferência, SEMPRE extrair, independente do tamanho:
   - Correções: "não sou", "na verdade", "na verdade sou", "me chama de", "não é isso", "errado", "está errado", "quero que me chame"
   - Preferências: "prefiro", "gostaria que", "não gosto", "me incomoda", "pode ser mais"
   - Essas palavras-chave têm prioridade sobre o filtro de tamanho.

4. **Comprimento da resposta do agente** — Se a resposta do agente também é trivial (< 50 chars), provavelmente não há nada de novo para aprender.

5. **Ratio de repetição no histórico** — Se as últimas N mensagens do usuário são muito similares à atual (>80% overlap de palavras), pular (já extraído antes).

### Mudança 2: Aplicar o gate nos 4 trigger points

Em cada ponto de trigger, adicionar verificação antes de disparar a extração:

```python
from app.services.memory_gate import should_extract_memories

if should_extract_memories(message, agent_response, history):
    asyncio.create_task(extract_and_save_memories(...))
```

**Arquivos:**
- `backend/app/worker/tasks.py` — 2 chamadas (linhas ~392 e ~1626)
- `backend/app/orchestrator/supervisor.py` — 1 chamada (linha ~454)
- `backend/app/orchestrator/graph.py` — 1 chamada (linha ~338)

### Mudança 3: Gate adicional para self-correction

A self-correction (`extract_agent_self_corrections`) pode ter um gate ainda mais restritivo:
- Só chamar se a resposta do agente tem mais de 100 caracteres (respostas curtas não violam regras complexas)
- Só chamar se o agent tem `training_memory_enabled=True` (não apenas `vector_memory_enabled`)
