# Backup & Restore Feature - Plan

## Overview

Add a backup and restore feature to allow saving and restoring the entire system state (PostgreSQL database, Weaviate vector data, and local files) via the frontend. The feature will be accessible from the "Configurações de IA" page, opening in a new tab.

---

## Architecture Decisions

### Tradeoffs

1. **Backup Format**: JSON-based export vs binary dump (pg_dump)
   - **Decision**: Use JSON-based export for portability across different database systems and easier inspection. Binary dumps are faster but tied to PostgreSQL version.
   - **Reasoning**: The system may be deployed with different database backends in the future. JSON is universal and human-readable for debugging.

2. **Backup Scope**: Full system vs selective tables
   - **Decision**: Full system backup including all PostgreSQL tables, Weaviate collections, and local files (documents, VFS).
   - **Reasoning**: The user wants to "salvar tudo" - partial backups add complexity that isn't needed initially.

3. **Execution Model**: Synchronous vs background job
   - **Decision**: Use background jobs for both backup and restore operations.
   - **Reasoning**: Backup/restore can take minutes for large datasets. Synchronous requests would timeout.

4. **Restore Strategy**: Full overwrite vs merge
   - **Decision**: Full overwrite - restore replaces all existing data.
   - **Reasoning**: Merge logic is complex and error-prone. The use case is migrating to a new system or restoring from a known state.

5. **File Storage**: Store backup files on server vs stream to client
   - **Decision**: Stream to client for download; require upload for restore.
   - **Reasoning**: Avoids server storage management and security concerns. User controls where backups are saved.

## Backend Implementation

### 1. New API Module: `backup.py`

**Location**: `backend/app/api/backup.py`

**Endpoints**:

#### `POST /backup/create`
- Triggers full system backup
- Returns JSON stream for download
- Uses background task for large datasets

**Response**: File download with `Content-Disposition: attachment; filename="basile_backup_YYYYMMDD_HHMMSS.json"`

**Backup JSON Structure**:
```json
{
  "metadata": {
    "version": "1.0.0",
    "created_at": "2026-04-04T00:00:00Z",
    "database_url": "postgresql://...",
    "weaviate_url": "http://..."
  },
  "tables": {
    "agents": [...],
    "mcps": [...],
    "documents": [...],
    ...
  },
  "weaviate": {
    "DocumentChunk": [...],
    "InformationBase": [...],
    "VectorMemory": [...]
  },
  "files": {
    "documents": {
      "uuid-filename.pdf": "<base64-encoded-content>",
      ...
    },
    "vfs": {
      "kb-uuid/file.md": "<base64-encoded-content>",
      ...
    }
  }
}
```

#### `POST /backup/restore`
- Accepts multipart form with backup JSON file
- Optional: `skip_files=true/false` query param
- Validates backup structure
- Clears existing data (in transaction where possible)
- Restores all tables, Weaviate collections, and files
- Returns status and count of restored items

**Response**:
```json
{
  "status": "success",
  "restored": {
    "tables": {"agents": 5, "mcps": 3, ...},
    "weaviate": {"DocumentChunk": 150, ...},
    "files": {"documents": 10, "vfs": 5}
  }
}
```

### 2. Backup Service: `backup_service.py`

**Location**: `backend/app/services/backup_service.py`

**Functions**:

- `create_full_backup(db: AsyncSession) -> bytes`
  - Exports all tables using SQLAlchemy reflection
  - Exports Weaviate collections via weaviate-client
  - Reads local files from UPLOAD_DIR and VFS root
  - Assembles JSON structure
  - Returns serialized JSON bytes

- `restore_full_backup(db: AsyncSession, backup_data: dict, skip_files: bool = False) -> dict`
  - Validates backup version compatibility
  - Clears all tables (respecting foreign key order)
  - Restores tables in dependency order
  - Clears and rebuilds Weaviate collections
  - Writes files to local filesystem
  - Returns restore statistics

### 3. Register Router

**File**: `backend/main.py`

Add import:
```python
from app.api import backup
```

Add router registration:
```python
app.include_router(backup.router, prefix="/backup", tags=["Backup"])
```

## Database Considerations

### Table Export Order (respecting foreign keys)

**Export order** (parents first):
1. `ai_providers`
2. `emotional_profiles`
3. `agent_groups`
4. `mcp_groups`
5. `skill_groups`
6. `vfs_knowledge_bases`
7. `information_bases`
8. `agents`
9. `mcps`
10. `skills`
11. `documents`
12. `vfs_files`
13. `agent_configs`
14. `agent_collaborators`
15. `pending_approvals`
16. `webhook_configs`
17. `job_logs`
18. `conversation_messages`
19. `api_keys`
20. Association tables (last)

**Import order** (reverse of export, children first cleared, then parents restored first):
1. Clear association tables first
2. Clear leaf tables
3. Clear parent tables last
4. Restore in export order above

### Weaviate Collections

Collections to backup:
- Document chunks (class name varies by document)
- Information base entries
- Vector memory entries

Each object's UUID, vector, and properties must be preserved.

### File Paths

- Documents: `UPLOAD_DIR` (from `app/services/document_processor.py`)
- VFS files: VFS root directory (from `app/services/vfs_rag_service.py`)

## Error Handling

### Frontend
- Network errors: Show snackbar with error message
- Invalid backup file: Show error dialog with validation details
- Restore failure: Show detailed error with rollback confirmation

### Backend
- Invalid backup format: HTTP 400 with validation errors
- Weaviate connection failure: HTTP 503
- Database transaction failure: Rollback all changes, HTTP 500
- File I/O errors: HTTP 500 with path details
- Memory limits for large backups: Stream JSON instead of building in memory

## Security Considerations

1. **Authentication**: Backup/restore endpoints require valid Bearer token (handled by existing axios interceptor)
2. **Authorization**: Consider adding admin-only restriction (new `access_level` check)
3. **File validation**: Validate uploaded backup JSON structure before processing
4. **SQL injection**: Use parameterized queries (already standard in codebase)
5. **Path traversal**: Sanitize file paths when restoring to prevent writing outside UPLOAD_DIR/VFS root

## Design Decisions (User-Confirmed)

1. **API Keys in Backup**: **Excluded by default** for security. Sensitive fields (`api_key`, `access_token`) will be stripped from backup. Users must reconfigure after restore.

2. **Large Backup Handling**: **Direct streaming** approach. JSON will be streamed to client without building entire structure in memory. Uses `StreamingResponse` in FastAPI.

3. **Dry Run Mode**: **Will be implemented**. Restore endpoint will accept `?dry_run=true` query param to preview what will be restored without modifying data.

4. **Backup History**: Not planned. Files are downloaded to client, not stored on server.

## Updated Backend Implementation Details

### Backup Endpoint Refinements

**Sensitive Fields to Exclude**:
- `ai_providers.api_key`
- `webhook_configs.access_token`
- `api_keys.key_hash`

These fields will be set to `null` in the backup JSON.

### Restore Endpoint Updates

**New Query Parameters**:
- `dry_run=true/false` - Preview restore without applying changes
- `skip_files=true/false` - Skip file restoration (database only)

**Dry Run Response**:
```json
{
  "dry_run": true,
  "would_restore": {
    "tables": {"agents": 5, "mcps": 3, ...},
    "weaviate": {"DocumentChunk": 150, ...},
    "files": {"documents": 10, "vfs": 5},
    "warnings": ["API keys were excluded from backup and will remain unchanged"]
  }
}
```

### Streaming Implementation

**Backend** (`backup.py`):
```python
from fastapi.responses import StreamingResponse
import json

async def backup_generator():
    yield '{"metadata": {...}, "tables": {'
    for table_name in tables:
        yield f'"{table_name}": {json.dumps(rows)},'
    yield '}, ...}'

return StreamingResponse(backup_generator(), media_type="application/json")
```

This avoids loading entire backup into memory for large datasets.

- User can navigate to "Configurações de IA" and access backup/restore
- User can click "Iniciar Backup" and download a `.json` file
