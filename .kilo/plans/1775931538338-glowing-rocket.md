# Plano: Agente Pensador (Thinker) - Tipo Separado de Agente

## Objetivo
Criar um **novo tipo de agente** chamado "thinker" (pensador) que funciona de forma independente do orquestrador. O pensador terá visão completa de TODOS os agentes da rede de colaboração para criar um planejamento estratégico antes da execução.

## Conceito Reformulado

### Taxonomia de Agentes

| Tipo | Função |
|------|--------|
| **normal** | Agente de tarefa simples - executa ação direta |
| **orchestrator** | Coordena colaboradores conectados a ele |
| **thinker** (NOVO) | Planejamento estratégico global - visão de toda a rede |

### Responsabilidades do Pensador

O pensador é como um "arquiteto" do sistema:
1. **Análise Global** - conhece todos os agentes disponíveis
2. **Decomposição** - quebra a tarefa em passos
3. **Alocação** - decide qual agente faz cada coisa
4. **Ordenação** - define sequência ideal
5. **Contexto** - passa informações necessárias

## Análise do Sistema Atual

### Arquitetura Existente
1. **Orquestrador (`AgentOrchestrator`)**: Decide quais colaboradores acionar com base na mensagem do usuário
   - Usa `gpt-4o-mini` para tomar decisões de colaboração
   - Método `should_collaborate()` analisa se colaboração é necessária
   
2. **Colaboradores**: Agentes especializados que executam tarefas específicas
   - Respondem ao orquestrador com formato estruturado

3. **Agente Planner Já existe**:
   - Campo `is_planner` no modelo `Agent` (linha 106)
   - Campo `planner_prompt` e `planner_model` (linhas 107-108)
   - Já utilizado em `tasks.py` (~linha 851)

### Problema Atual
- O orquestrador gast tokens para "pensar" sobre qual estratégia usar
- Decisões de colaboração + planejamento são feitas pelo mesmo modelo leve
- Não há separação clara entre planejamento estratégico e execução

## Proposta: Agente Pensador com Two-Stage Execution

### Arquitetura Proposta

```
┌─────────────────────────────────────────────────────────────────┐
│                      USER REQUEST                                │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│              THINKER/PLANNER AGENT (Modelo Forte)                 │
│  - Analisa a tarefa completa                                     │
│  - Define estratégia de execução                                │
│  - Identifica quais agentes acionar                             │
│  - Determina ordem de execução                                  │
│  - Output: Plano de execução estruturado                       │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│              EXECUTOR/ORCHESTRATOR (Modelo Leve)                 │
│  - Recebe o plano do pensador                                    │
│  - Executa as tarefas conforme plano                            │
│  - Delega para colaboradores                                    │
│  - Retorna resultado ao usuário                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Opções de Implementação

#### Opção 1: Chamada Prévia (Pre-Planning)
O agente pensador é chamado **antes** do orquestrador principal:
- O pensador analisa a requisição e retorna um plano
- O orquestrador executa o plano
- Vantagem: Separação clara de responsabilidades
- Desvantagem: Mais uma chamada de API

#### Opção 2: Advisor Tool Pattern (Integração Nativa)
O pensador é chamado **durante** a execução quando necessário:
- O executor pode chamar o pensador quando enfrenta complexidade
- Similar ao padrão da Anthropic
- Desvantagem: Requer suporte nativo da API

#### Opção 3: Híbrido - Planner Mode do Agente
Usar o campo `is_planner` existente no modelo:
- O agente pode operar em modo "planejador"
- Gera um plano estruturado antes da execução
- Orquestrador usa o plano gerado
- Vantagem: Já tem infraestrutura parcial

### 5. Integração no Orquestrador

Quando um orchestrator tem um thinker vinculado:
1. Antes de qualquer execução, chama o thinker
2. O thinker retorna o plano
3. O orquestrador segue o plano literalmente
4. Não toma decisões próprias (apenas executa)

## Perguntas para Definição

### 1. Associação
Um thinker pode ser associado a:
- a) Apenas UM orchestrator (simples)
- b) MÚLTIPLOS orchestrators (flexível)
- c) Qualquer agente que tenha colaboradores

### 2. Ativação
O thinker é invoked:
- a) **Sempre** antes de tudo (pré-obrigatório)
- b) Por config do orchestrator (toggle)
- c) Por palavra-chave na mensagem

### 3. Modelo
Qual modelo usar para thinker:
- a) Mesmo do orchestrator (simples)
- b) Modelo mais forte (O4, Opus, etc) (inteligente)
- c) Deixar configurável por organização

### 4. Fallback
Se thinker falhar:
- a) Retorna erro ao usuário
- b) Usa lógica original do orchestrator
- c) Tenta novamente com prompt simplificado

## Decisões Tomadas

- **Associação**: Qualquer agente que tenha colaboradores pode ter um thinker vinculado
- **Ativação**: Por palavra-chave na mensagem do usuário

### Configuração de Ativação

O thinker é ativado quando a mensagem contém trigger keywords:
- Palavras estratégicas: "planejar", "pensar", "analisar", "estratégia"
- Palavras de decomposição: "passo a passo", "sequência", "processo"
- Palavras de planejamento: "como fazer", "melhor forma", "abordagem"

### Decisões Finais

- **Modelo**: Usar configuração normal do agente thinker (mesmo sistema de modelos)
- **Fallback**: Se thinker falhar, usar lógica original do agente

### Múltiplos Thinkers - Planejamento Específico vs Global

O sistema pode ter **MÚLTIPLOS pensadores**, cada um com sua especialização:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AGENTES THINKER DISPONÍVEIS                   │
├─────────────────────────────────────────────────────────────────┤
│  Thinker 1: "Planejador Global"                                 │
│  ├─ trigger_keywords: ["planejar", "estratégia", "visao"]      │
│  └─ Função: Planeja operação completa, visão holística         │
│                                                                 │
│  Thinker 2: "Especialista em Dados"                            │
│  ├─ trigger_keywords: ["dados", "consulta", "buscar"]          │
│  └─ Função: Foco em planejamento de queries e operações DB      │
│                                                                 │
│  Thinker 3: "Especialista em流程"                               │
│  ├─ trigger_keywords: ["processo", "fluxo", "automação"]        │
│  └─ Função: Foco em automação e workflows                      │
└─────────────────────────────────────────────────────────────────┘
```

#### Tipo de Ativação

| Tipo | Descrição | Exemplo |
|------|-----------|---------|
| **Específico** | Pensador chamado por keyword específica | "dados" → Thinker de Dados |
| **Global** | Pensador chamado para visão completa | "estratégia" → Thinker Global |

#### Como Funciona

1. **Agente normal** pode ter **múltiplos thinkers vinculados**
2. Cada thinker tem seus próprios **trigger_keywords** (como agente normal)
3. Quando a mensagem contém um keyword → chama o thinker correspondente
4. **Múltiplos thinkers podem ser chamados** se suas keywords corresponderem
5. Os planos são agregados e executados em sequência

#### Exemplo de Associação

```
Agente Atendente (normal)
  ├─ Thinker Dados (keywords: "dados", "buscar", "consultar")
  ├─ Thinker Global (keywords: "planejar", "estratégia")
  └─ Thinker Fluxo (keywords: "processo", "automação")
```

Se usuário digitar "me ajuda a planejar os dados do cliente" → Chama **Thinker Global + Thinker Dados** (ambos têm keywords correspondentes)

Se usuário digitar "buscar dados do cliente" → Chama **Thinker Dados** (keyword "buscar")

Se usuário digitar "planejar automação de processos com dados" → Chama **Todos os 3** (todas keywords correspondem)

Se usuário digitar "me ajuda a planejar os dados do cliente" → Chama **Thinker Global + Thinker Dados** (ambos têm keywords correspondentes)

Se usuário digitar "buscar dados do cliente" → Chama **Thinker Dados** (keyword "buscar")

Se usuário digitar "planejar automação de processos com dados" → Chama **Todos os 3** (todas keywords correspondem)
